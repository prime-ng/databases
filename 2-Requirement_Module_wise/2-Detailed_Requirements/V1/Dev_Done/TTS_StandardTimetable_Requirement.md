## 1. EXECUTIVE SUMMARY

### 1.1 Purpose

The StandardTimetable module is a lightweight, read-focused companion to the SmartTimetable module. It provides a manual timetable builder interface (drag-and-drop placement grid) and standardized read-only views of the published timetable organized by class, teacher, and room. It consumes data generated and stored by the SmartTimetable module, adding no independent data pipeline of its own.

### 1.2 Scope

StandardTimetable covers: (a) a manual placement interface with four tabs (Drag & Drop Grid, Class Wise, Teacher Wise, Room Wise), (b) read-only standard views of the published timetable filtered by class, teacher, or room, (c) display of the timetable grid using the shared `analytics/reports/_grid.blade.php` partial from SmartTimetable. The module has no models, no services, no FormRequests, and no tests. It has a single controller with one method (`manualPlacement()`).

## 2. MODULE OVERVIEW

### 2.1 Business Purpose

School administrators, class teachers, and the principal need to view the current active timetable in a format that is quick to read and filtered to their context:

- A class teacher wants to see their class's full weekly timetable
- A subject teacher wants to see all their assigned periods across classes and days
- The admin wants to see room occupancy to plan events or manage room changes

StandardTimetable provides these standard read views on top of the data generated by SmartTimetable. It also provides a manual placement interface for schools that prefer to build their timetable using drag-and-drop rather than the automated solver.

### 2.2 Relationship to SmartTimetable

StandardTimetable is a thin display layer over SmartTimetable:

```
SmartTimetable (data owner)
  → tt_timetable, tt_timetable_cell, tt_timetable_cell_teacher
  → tt_activity, tt_teacher_workload, tt_room_utilizations
  → AnalyticsService (getClassReport, getTeacherReport, getRoomReport)
  → _grid.blade.php partial

StandardTimetable (display layer)
  → Calls AnalyticsService methods
  → Renders _grid partial
  → Provides manual placement UI for drag-drop scheduling
```

### 2.3 User Personas

| Persona | Primary Actions |
|---|---|
| School Admin / Timetable Coordinator | Manual placement, all three standard views |
| Principal | Class, teacher, and room views |
| Class Teacher | Class-wise view (own class) |
| Subject Teacher | Teacher-wise view (own schedule) |
| Parent Portal (future) | Class-wise view of published timetable |

## 3. BUSINESS RULES

**BR-TTS-001:** Only a PUBLISHED `tt_timetable` is displayed in any StandardTimetable view. Draft, generated, or approved-but-unpublished timetables are not visible.

**BR-TTS-002:** StandardTimetable views are read-only. No cell modification is allowed from this module. All editing is performed through SmartTimetable's Refinement module.

**BR-TTS-003:** The manual placement interface allows drag-and-drop activity placement for schools using manual scheduling (i.e., `tt_timetable.generation_method = 'MANUAL'`). Activities placed manually create the same `tt_timetable_cell` records as auto-generated timetables.

**BR-TTS-004:** A user can only view the timetable data for their own tenant (enforced by stancl/tenancy).

**BR-TTS-005:** A teacher user can only see their own teacher-wise view. A class teacher can see their class's view. Admin/Coordinator can see all views.

**BR-TTS-006:** The module shares the `tt_*` table prefix and all underlying tables with SmartTimetable. No new tables are introduced by StandardTimetable.

## 4. FUNCTIONAL REQUIREMENTS

### FR-TTS-001: Standard View — Class Wise

**FR-TTS-001.1** The system must provide a class-wise timetable view showing the full weekly grid (days × periods) for a selected class and section.

**FR-TTS-001.2** Each cell must display: subject name, teacher name(s), room name/number, and period type (color coded by `tt_period_type.color_code`).

**FR-TTS-001.3** The system must allow filtering by: academic term, timetable type, class, and section.

**FR-TTS-001.4** The system must use the shared `analytics/reports/_grid.blade.php` partial from SmartTimetable module for rendering the grid. Data is sourced via `AnalyticsService::getClassReport(timetableId, classId, sectionId)`.

**FR-TTS-001.5** The grid must visually differentiate period types: teaching periods (normal), break/lunch (shaded), free periods (striped), exam periods (highlighted), locked cells (icon indicator).

**FR-TTS-001.6** The class-wise view must be accessible at `GET /standard-timetable/class-view` with query parameters `?class_id=&section_id=&timetable_id=`.

### FR-TTS-002: Standard View — Teacher Wise

**FR-TTS-002.1** The system must provide a teacher-wise timetable view showing all periods assigned to a selected teacher across the full week.

**FR-TTS-002.2** Each cell must display: class name, section, subject name, room number.

**FR-TTS-002.3** The system must highlight free periods (no assignment) and gap periods (periods between assignments in the same day).

**FR-TTS-002.4** Data is sourced via `AnalyticsService::getTeacherReport(timetableId, teacherId)`.

**FR-TTS-002.5** The teacher-wise view must be accessible at `GET /standard-timetable/teacher-view` with query parameters `?teacher_id=&timetable_id=`.

**FR-TTS-002.6** A teacher user must only be able to access their own teacher_id. Admin/Coordinator can select any teacher.

### FR-TTS-003: Standard View — Room Wise

**FR-TTS-003.1** The system must provide a room-wise timetable view showing all periods for which a selected room is booked.

**FR-TTS-003.2** Each cell must display: class+section, subject name, teacher name, activity type.

**FR-TTS-003.3** Free (unbooked) periods must be shown as empty, allowing admin to identify room availability.

**FR-TTS-003.4** Data is sourced via `AnalyticsService::getRoomReport(timetableId, roomId)`.

**FR-TTS-003.5** The room-wise view must be accessible at `GET /standard-timetable/room-view` with query parameters `?room_id=&timetable_id=`.

### FR-TTS-004: Manual Placement Interface

**FR-TTS-004.1** The manual placement page (`pages/manual-placement.blade.php`) must present a tabbed interface with four tabs: Drag & Drop Grid, Class Wise, Teacher Wise, Room Wise.

**FR-TTS-004.2** The Drag & Drop Grid tab must display an interactive weekly grid (days × periods) where activities can be dragged from an activity palette and dropped into period slots.

**FR-TTS-004.3** On drop, the system must validate: teacher not already assigned at that slot (TeacherConflictConstraint), room not already booked (RoomExclusiveUseConstraint), student group not double-booked (NotOverlappingConstraint). Violations must be shown as inline warning banners without blocking the drop.

**FR-TTS-004.4** On successful drop, the system must create a `tt_timetable_cell` record with `source = 'MANUAL'` and a `tt_timetable_cell_teacher` record for each assigned teacher.

**FR-TTS-004.5** Activities displayed in the palette must come from the active `tt_activity` records for the selected `academic_term_id` and `timetable_type_id`.

**FR-TTS-004.6** A placed activity must be removable by clicking a remove button on the cell, which soft-deletes the `tt_timetable_cell` record.

**FR-TTS-004.7** The grid must show a placement progress indicator (e.g., "45 of 120 activities placed").

**FR-TTS-004.8** The manual placement view is accessible at `GET /standard-timetable/manual-placement` (named route `standard-timetable.menu.manualPlacement`).

### FR-TTS-005: Timetable Selector

**FR-TTS-005.1** All standard views must present a timetable selector dropdown listing all PUBLISHED timetables for the current academic session, ordered by `published_at DESC`.

**FR-TTS-005.2** The most recently published timetable must be pre-selected by default.

**FR-TTS-005.3** The selector must show the timetable code, name, version, and published_at date.

### FR-TTS-006: Print and Export

**FR-TTS-006.1** Each standard view must provide a browser print button that triggers `window.print()` with a print-optimized CSS stylesheet (landscape orientation, hidden navigation).

**FR-TTS-006.2** The class-wise view must support a simple CSV export of the grid data for the selected class/section.

**FR-TTS-006.3** PDF export via DomPDF is planned (not yet implemented) — the view structure must be compatible with blade-to-PDF rendering.

## 5. DATA MODEL

### 5.1 Overview

StandardTimetable introduces no new database tables. All data is read from the `tt_*` tables owned by SmartTimetable. The key tables consumed are:

| Table | Usage |
|---|---|
| `tt_timetable` | Select PUBLISHED timetable; show metadata |
| `tt_timetable_cell` | Read cell grid (day_of_week, period_ord, activity_id, room_id) |
| `tt_timetable_cell_teacher` | Read teacher assignments per cell |
| `tt_activity` | Read activity details (subject, class, section, study_format) |
| `tt_teacher_workload` | Read teacher load analytics for teacher-wise view |
| `tt_resource_booking` | Read room bookings for room-wise view |
| `tt_period_set_period_jnt` | Read period structure (start_time, end_time, period_type) |
| `tt_period_type` | Read period type colors and labels |
| `tt_generation_run` | Display latest run stats for informational header |
| `tt_conflict_detection` | Show conflict count badge if any unresolved conflicts exist |
| `sch_classes` | Class names for display |
| `sch_sections` | Section names for display |
| `sch_rooms` | Room names/numbers for display |
| `sch_teachers` | Teacher names for display |
| `sch_subjects` | Subject names for display |

### 5.2 Data Access Pattern

StandardTimetable calls SmartTimetable's `AnalyticsService` methods:

```php
// Class-wise view
AnalyticsService::getClassReport(int $timetableId, int $classId, int $sectionId): array

// Teacher-wise view
AnalyticsService::getTeacherReport(int $timetableId, int $teacherId): array

// Room-wise view
AnalyticsService::getRoomReport(int $timetableId, int $roomId): array
```

Each method returns a structured array consumed by the shared `analytics/reports/_grid.blade.php` partial.

## 8. WORKFLOW

### 8.1 Standard View Access Flow

```
User navigates to Standard Timetable
  → Timetable selector loads PUBLISHED timetables
  → User selects timetable (or latest is pre-selected)
  → User selects view type: Class / Teacher / Room
  → User selects entity (class+section / teacher / room)
  → System calls AnalyticsService.get[Class|Teacher|Room]Report()
  → _grid partial renders day × period grid
  → User can print or export CSV
```

### 8.2 Manual Placement Flow

```
Admin creates a new MANUAL timetable record (tt_timetable, generation_method=MANUAL)
  → Navigate to Standard Timetable → Manual Placement
  → Drag-and-Drop tab active
  → Activity palette loads unplaced activities
  → Admin drags activity card → drops onto period cell
    → Conflict check (teacher, room, group)
      ├── Conflict: show warning, allow or cancel drop
      └── No conflict: create tt_timetable_cell + tt_timetable_cell_teacher
  → Repeat until all activities placed
  → Click [Validate] → PreGenerationValidation check
  → Click [Submit for Approval] → timetable.status → GENERATED
  → Normal approval/publish flow proceeds (handled by SmartTimetable)
```

## 12. ERROR HANDLING

### 12.1 No Published Timetable

If no PUBLISHED timetable exists, all standard views must display an informational message: "No published timetable found. Please generate and publish a timetable first." with a link to SmartTimetable generation.

### 12.2 Analytics Not Computed

If `tt_teacher_workload` / `tt_room_utilizations` are empty (analytics not yet computed), the system must show a placeholder message and trigger lazy computation if the user is an Admin.

### 12.3 Manual Placement Conflict

When a drop creates a conflict in manual placement:
- The cell is highlighted in red with a conflict icon
- A tooltip shows the conflict description
- The placement is saved (allow with warning), not blocked
- The conflict is recorded in `tt_timetable_cell.has_conflict = 1` and `conflict_details_json`
